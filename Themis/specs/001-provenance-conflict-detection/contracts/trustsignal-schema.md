# TrustSignal Schema

## Purpose

Defines the structure of the TrustSignal output generated by the THEMIS evaluation service.

## TrustSignal Object

```json
{
  "score": 0.0,
  "flags": ["DATE_CONFLICT", "VERSION_MISMATCH", "COMPLIANCE_RISK"],
  "evidence": [
    {
      "source_id": "string",
      "lineage": ["string"],
      "conflicts": [
        {
          "doc_a": "string",
          "doc_b": "string",
          "conflicting_claim": "string",
          "claim_a": "string",
          "claim_b": "string",
          "conflict_type": "string",
          "rule_id": "string",
          "evidence": ["string"]
        }
      ],
      "rule_matches": [
        {
          "rule_id": "string",
          "description": "string",
          "trigger_conditions": "string",
          "evidence": ["string"]
        }
      ]
    }
  ],
  "lineage_chains": [
    {
      "chain_id": "string",
      "nodes": [
        {
          "id": "string",
          "version": "string",
          "author": "string",
          "date": "string",
          "source_type": "string",
          "regulatory_domain": "string"
        }
      ]
    }
  ],
  "source_scores": [
    {
      "source_id": "string",
      "score": 0.0,
      "reason": "string"
    }
  ]
}
```

## Field Definitions

- `score`: Composite trust score between 0 and 1.
- `flags`: Conflict and compliance labels detected during evaluation.
- `evidence`: Per-source evidence payload with lineage, conflicts, and rule matches.
- `lineage_chains`: Explicit provenance chains for document versions.
- `source_scores`: Individual trust scores for each source and the reasoning behind them.

## Guidelines

- Each `conflict` entry must reference both documents and the canonical claim key.
- Each `rule_match` must include a rule identifier and supporting evidence.
- `lineage_chains` must preserve the order of versions from origin to latest.
- `score` must never appear without `evidence`.
